Is PgFleet production-ready, and who is it for?

PgFleet is aimed at small teams, solo operators, and infrastructure-minded developers who want managed-Postgres ergonomics on hardware they control. It bundles the operational essentials — provisioning, WAL archiving, scheduled backups, point-in-time recovery, replication, and failover — into one control plane.

As with any self-hosted database platform, "production-ready" depends on you: validate your backups with restore drills, configure replicas and alerting, and rehearse failover before you depend on it. PgFleet gives you the tooling to do all of that; the discipline of running it is yours.

How is it different from AWS RDS or Crunchy Bridge?

RDS and Crunchy Bridge are fully managed cloud services: the vendor owns the hardware, the control plane, and ultimately your data location, and you pay a recurring per-instance premium for that convenience.

PgFleet is the opposite trade-off. You run the control plane yourself with Docker, your Postgres instances live on your hosts, and your backups go to your S3 or MinIO bucket. You get a comparable operational experience — one-click provisioning, automatic backups, PITR, failover, observability — with no vendor lock-in and no per-instance bill. The cost is that you operate the host and object storage.

Where are backups stored, and are they actually restorable?

Backups go to an object store you control — any S3-compatible service, including MinIO for fully on-prem setups. PgFleet uses pgBackRest for full and differential backups together with continuous WAL archiving.

The only backup you can trust is one you have restored. PgFleet lets you restore a backup into a fresh instance as a drill, so you can confirm it is genuinely restorable rather than assuming it is. Make those drills routine.

How does point-in-time recovery (PITR) work?

PITR combines a base backup with the continuous stream of archived WAL. To recover, PgFleet starts from the most recent base backup taken before your target time, then replays archived write-ahead logs up to the exact second you choose.

A guided wizard walks you through picking the target timestamp and target instance, so you can rewind to the moment just before a bad deploy or an accidental DELETE — anywhere inside your retention window — without hand-writing recovery configuration.

Are backups encrypted at rest?

Yes — turn on backup encryption and PgFleet has pgBackRest cipher every backup and archived WAL file with AES-256-CBC before it leaves the host. The data is encrypted in the repository itself, so it stays protected regardless of what the bucket does, and a leaked object is useless without the key.

This is independent of (and stacks with) bucket-side server-side encryption and host-volume encryption. Keep the encryption key in your own secret store; PgFleet never ships it off your environment.

Can I name or annotate a backup?

Yes. When you take an on-demand backup you can attach an optional note or annotation — for example “pre-migration” or “before v2 deploy”. The note rides with the backup in pgBackRest's metadata and shows up in the backup list, so a set of backups is self-documenting instead of an anonymous wall of timestamps.

You can also pick the backup type — full, differential, or incremental — at the moment you take it, and delete a backup you no longer need from the same view.

What is a delta restore?

A delta restore is an option on point-in-time recovery that only rewrites the files that actually differ between the existing data directory and the backup, instead of wiping and recreating everything. pgBackRest compares checksums and copies just the changed blocks.

On a large database that has drifted only a little from a recent backup, that makes the restore dramatically faster and lighter on I/O — useful for re-syncing an instance or rehearsing a recovery without paying for a full rebuild. For a clean target you can still do a normal full restore.

Can I run a second backup repository for a 3-2-1 strategy?

Yes. PgFleet supports a second pgBackRest repository (repo2) alongside the primary one, so the same backups and WAL are written to two independent destinations. Point repo2 at different media or a different provider and you have 3-2-1 built in: three copies, on two kinds of media, one of them off-site.

For stronger durability, target an object-lock (WORM) bucket so the second copy can't be altered or deleted within its retention window — which keeps a clean recovery point even if the primary repo is compromised.

Does the host port or IP change when I stop and start an instance?

It can. Managed instances publish on ephemeral host ports assigned by Docker, so the host-side port may differ after a stop/start. This is expected behavior, not a bug.

PgFleet handles it for you: on every start it re-reads the published port from Docker and persists it, and after a control-plane restart the reconciler re-adopts running containers and refreshes the recorded port. The dashboard therefore always shows the port the instance is currently reachable on.

For an address that never moves, front the instance with a reverse proxy or a Cloudflare Tunnel and connect your app to that stable hostname — see the next question. There is more detail on the features page.

How do I expose an instance to the internet safely?

Instances are private by default, bound to 127.0.0.1 so nothing is reachable from outside the host until you decide otherwise. Keep it that way unless you have a concrete reason to publish.

When you do need remote access, do not expose the raw Postgres port to the world. Put a reverse proxy or a Cloudflare Tunnel in front of it — the tunnel also gives you a stable hostname even as the origin port changes — and place an identity provider such as Authelia (or any OIDC IdP) in front of human-facing surfaces so only authenticated users get through. The public/private toggle makes the intent explicit per instance.

Is the admin dashboard protected, and can I use SSO?

Yes. Out of the box, PgFleet uses its own email + password login with a bootstrap admin seeded on first run — no external dependency required. The dashboard is a control plane for your whole fleet, so it should never sit open on the internet regardless.

Single sign-on is optional and toggled by environment. Set PGFLEET_SSO_EMAIL_HEADER on the API and NEXT_PUBLIC_OIDC_ENABLED=1 on the dashboard, and a “Sign in with SSO” button appears: PgFleet then trusts a forward-auth proxy (Authelia or any OIDC proxy) to authenticate, and maps the user’s groups to PgFleet admin / operator / viewer roles. Leave those unset and SSO is simply disabled. See the next question and the getting-started guide.

How exactly do I enable or disable SSO?

Disabled (default): do not set PGFLEET_SSO_EMAIL_HEADER. PgFleet mounts no SSO endpoint and the login page only offers built-in email + password.

Enabled: on the API set PGFLEET_SSO_EMAIL_HEADER (e.g. Remote-Email) plus the group-mapping vars PGFLEET_SSO_GROUPS_HEADER, PGFLEET_SSO_ADMIN_GROUP, and PGFLEET_SSO_OPERATOR_GROUP; on the dashboard set NEXT_PUBLIC_OIDC_ENABLED=1. PgFleet then trusts the email/groups headers injected by your proxy and issues a normal session.

Two invariants keep this safe: the proxy must strip client-supplied copies of those headers before forward-auth, and the API must be reachable only through the proxy (never published directly). Authelia adds MFA (TOTP) and brute-force regulation in front, and disabled PgFleet accounts are still rejected at the SSO exchange so off-boarding revokes access. Password login remains as a fallback.

Can I back up and restore a remote database I don't manage?

Yes — that is the remote backup/restore feature. Enter the host, port, and credentials of an external Postgres database and PgFleet can back it up to your object store, even though it does not run that database.

You can also restore a remote backup into a managed instance, which makes PgFleet a practical migration path: adopt the external database, verify the restore against a clone, then cut over when you are confident. See Remote backup & restore.

How does automatic failover work — can it cause split-brain?

PgFleet ships in-house automatic failover — no external orchestrator required. A quorum guard sits in front of every promotion: a replica is only promoted when a majority of observers agree the primary is genuinely gone, so a transient network blip or a brief GC pause never triggers a failover. PgFleet then fences and removes the old primary first, and only then promotes a healthy replica.

Because the quorum prevents a premature promotion and the failed primary is fenced before any new writer comes online, you never end up with two nodes accepting writes — no split-brain. The PgCat router then directs traffic to the newly promoted primary so applications follow the failover transparently.

What Postgres versions and extensions are supported?

PgFleet provisions Postgres from standard Docker images, so it tracks the current, actively-supported major versions of Postgres; you pick the version when you create an instance. For time-series workloads it offers first-class TimescaleDB support so you can run hypertables out of the box.

Because instances are real Postgres containers, you keep the flexibility of the wider Postgres extension ecosystem rather than a locked-down managed subset.

How do I run recovery drills to know my backups are good?

Restore into a fresh, throwaway instance and check the data — that is the drill. Because provisioning and restore are one-click operations, you can rehearse recovery on a schedule without risking the original instance.

Combine this with PITR to practice rewinding to a specific timestamp, and with one-click clone to test changes against a real copy. A backup you have never restored is a hope, not a backup.

How is my data secured and encrypted at rest?

Because PgFleet is self-hosted, encryption at rest is something you own end to end and can satisfy at the layers that matter: enable encryption at rest on the host volumes backing your instances, and turn on server-side encryption on your S3 or MinIO bucket so archived WAL and backups are encrypted in object storage.

For data in transit, terminate TLS at your reverse proxy or Cloudflare Tunnel, and keep instances private by default so nothing is exposed unencrypted. Credentials for remote databases and object storage stay within your own environment.

How do I monitor PgFleet and my instances?

The dashboard gives you live and historical analytics — throughput, connections, replication lag — plus query insights for slow statements and streamed logs, so you can see what every instance is doing without leaving the browser.

On top of that, configure alerts on health, replication lag, disk, and backup status so issues reach you proactively. The metrics are well-suited to wiring into your existing Prometheus-based monitoring and dashboards alongside the built-in views.

What does the PgCat router actually do for me?

PgCat is a connection pooler and router that sits in front of an instance and its replicas. It gives your application a single endpoint while splitting writes to the primary and reads across replicas with load-balancing, and it pools connections so a burst of clients does not overwhelm Postgres. You choose the pool mode: transaction mode multiplexes many clients onto few server connections, while session mode holds a connection for a client's whole session.

It is not a black box: the cluster page shows live pool stats from PgCat's own SHOW POOLS and SHOW STATS — active and waiting clients, server connections, and per-pool throughput. The router also auto-bans an unhealthy backend so one sick replica stops receiving queries, can optionally mirror traffic to a shadow backend, and follows failovers: after PgFleet promotes a new primary, writes are repointed automatically with no app reconfiguration.

What are the licensing and self-hosting terms?

PgFleet is open-source and self-hosted. You run the entire stack on your own infrastructure — control plane, Postgres instances, and object storage — with no usage-based fees and no vendor lock-in.

For the authoritative license terms, check the LICENSE file in the GitHub repository, which always reflects the current licensing for the version you are running.

What do I need to run it, and how do I get started?

You need a host with Docker and an S3-compatible object store (managed S3 or a self-hosted MinIO). The control plane itself is a Go backend with a Next.js dashboard, packaged to run with Docker.

Clone the repository, start the control plane, point it at your bucket, and provision your first instance from the dashboard. From there, enable scheduled backups, add a replica, and run a restore drill to confirm everything is wired up. Follow the step-by-step getting-started guide, or jump straight to the GitHub repo.
