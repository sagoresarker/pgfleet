Features

A complete, self-hosted Postgres platform.

PgFleet bundles the operational machinery you would otherwise stitch together by hand — provisioning, durable backups, recovery, high availability, observability, and day-two operations — behind a single control plane you run yourself.

01

Provisioning & lifecycle

Create and manage Postgres instances as easily as any cloud database service.

One-click provisioning

Launch a Postgres instance in Docker — pick the version, extensions, and resources — without touching a compose file.

Full lifecycle control

Start, stop, restart, and decommission instances from the dashboard, with the control plane reconciling actual container state.

One-click clone

Fork a fresh, independent copy of any instance for staging, testing, or a safe migration rehearsal — without disturbing production.

02

Backups, PITR & disaster recovery

Durable off-host backups and recovery you can actually rehearse.

Automatic WAL archiving

Write-ahead logs stream continuously to S3 or MinIO, so every committed transaction is durable off the host within seconds.

Scheduled backups

Full and differential backups on a schedule via pgBackRest, with configurable retention — all written to your own object store.

Point-in-time recovery

A guided wizard restores to any second in your retention window by replaying archived WAL on top of the nearest base backup.

Recovery drills

Restore into a fresh instance to verify a backup is genuinely restorable — turning disaster recovery into a routine you can practice.

03

High availability

Streaming replication, automatic failover, and connection routing — in house.

Streaming replication

Attach hot-standby replicas that stream WAL from the primary, ready to serve reads or take over.

In-house automatic failover

PgFleet detects a failed primary, fences and removes it, then promotes a healthy replica — no external orchestrator required.

PgCat read/write router

A built-in PgCat layer gives a single connection endpoint that splits reads and writes and follows failovers transparently.

04

Observability

Know exactly what every instance is doing, right now and over time.

Live & historical analytics

Real-time throughput, connections, and replication lag, plus historical trends to spot regressions before they bite.

Query insights

Surface slow and frequent queries so you can find the statement that is dragging down a workload.

Logs

Stream Postgres and container logs straight from the dashboard, without shelling into the host.

Alerting

Raise alerts on health, replication lag, disk, and backup status so problems reach you before users do.

05

Operate

Day-two tooling that keeps you out of SSH sessions and ad-hoc scripts.

SQL console

Run queries against any instance directly from the browser, with results rendered inline.

Container exec

Open an interactive shell inside an instance container for the rare moment you need to get your hands dirty.

Public / private toggle

Each instance is private by default on 127.0.0.1; flip a switch to publish it deliberately when you mean to.

Logical dump download

Grab a portable pg_dump logical backup from the dashboard whenever you need an offline, restorable copy.

06

Remote backup & restore

Bring an external database under PgFleet's protection without re-platforming first.

Adopt an external DB

Enter the host, port, and credentials of a database you do not manage and have PgFleet back it up to your object store.

Restore into PgFleet

Restore a remote backup into a managed instance to complete a migration, with the control plane taking over from there.

Migrate without downtime drama

Rehearse the migration against a clone, verify the restore, then cut over — keeping the original safely in place until you are ready.

07

Extensions

First-class support for the extensions teams actually reach for.

TimescaleDB

Provision time-series–ready instances with TimescaleDB enabled for high-ingest, hypertable workloads out of the box.

08

Access & single sign-on

Built-in auth out of the box, optional forward-auth SSO when you want it.

Email + password by default

PgFleet ships with built-in user accounts and roles. A bootstrap admin is seeded once on an empty database — no external dependency needed to log in.

Optional SSO (forward-auth)

Set PGFLEET_SSO_EMAIL_HEADER on the API and NEXT_PUBLIC_OIDC_ENABLED=1 on the dashboard to add a “Sign in with SSO” button. PgFleet then trusts a forward-auth proxy (Authelia or any OIDC proxy) for identity.

Group → role mapping

The proxy’s verified groups map to PgFleet admin, operator, or viewer roles (defaulting to least-privilege viewer). Authelia adds MFA and brute-force regulation in front. Enable it →

What happens when an instance restarts (and the IP/port)

Managed instances publish on ephemeral host ports — the host port is assigned by Docker, not pinned by you. Each time an instance starts, PgFleet re-reads the published port from Docker and persists it, so the dashboard always shows the port the instance is currently reachable on.

Because of this, the host port can change across a stop/start. After a control-plane restart, the reconciler re-adopts every running container and refreshes the recorded port, so PgFleet's view stays in sync with reality rather than drifting to a stale value. The instance's internal identity, data, and configuration are unchanged — only the host-side port mapping may differ.

For a stable entry point that does not move when the origin port changes, put a reverse proxy or a Cloudflare Tunnel in front of the instance. The proxy or tunnel hostname is what your application connects to; it stays constant while PgFleet keeps the underlying origin port up to date behind it. In short: 127.0.0.1:<ephemeral-port> may change, but db.example.com does not.
