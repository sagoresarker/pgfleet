From clone to a recoverable instance.

A concise, credible quickstart. You will bring up the control plane and its dependencies with Docker Compose, set a handful of environment variables, log in, provision your first Postgres instance, and take a backup — then optionally put SSO in front of the dashboard.

1

Prerequisites

PgFleet provisions Postgres as Docker containers, so the host needs a working Docker engine. Everything else is brought up for you.

Docker Engine

A recent Docker that can pull postgres images and manage containers on your host.

Docker Compose v2

The bundled docker compose plugin — used to stand up the meta-DB and object store.

Object storage: backups need an S3-compatible bucket. The compose stack ships a local MinIO so you can try everything on one host; point it at managed S3 later by changing the PGFLEET_S3_* variables.

2

Clone and bring up the stack

Clone the repository and start the dependencies — the metadata database (Postgres) and the MinIO object store — with Docker Compose. These back the API and the Next.js dashboard.

shell 
$ git clone https://github.com/sagoresarker/pgfleet.git
$ cd pgfleet

# bring up meta-DB + MinIO (object store)
$ docker compose -f deploy/docker-compose.yml up -d

# meta-db → localhost:5433 · MinIO API → :9000 · console → :9001

With the dependencies healthy, start the API and dashboard (the Go control plane reads your .env; the Next.js UI talks to it):

shell 
# copy the sample env and edit the secrets (next step)
$ cp .env.example .env

# run the API + dashboard (loads .env)
$ make run

# dashboard → http://localhost:3000 · API → http://localhost:8080
3

Set the key environment variables

Edit .env before first run. The control plane refuses to start without the three secrets, and the bootstrap admin is seeded once on an empty database. Generate strong values rather than reusing the samples.

generate secrets 
# JWT secret — at least 32 bytes
$ openssl rand -hex 32

# master key — exactly 32 bytes, base64-encoded
$ openssl rand -base64 32
VariableWhat to set
PGFLEET_META_DB_DSNrequired DSN for the metadata Postgres, e.g. postgres://pgfleet:pgfleet@localhost:5433/pgfleet?sslmode=disable.
PGFLEET_JWT_SECRETrequired Signs session tokens. Use a random string of at least 32 bytes.
PGFLEET_MASTER_KEYrequired Encrypts stored credentials. Exactly 32 bytes, base64-encoded.
PGFLEET_BOOTSTRAP_ADMIN_EMAIL
PGFLEET_BOOTSTRAP_ADMIN_PASSWORD		required The first admin, seeded once on an empty meta-DB. This is your initial login.
PGFLEET_S3_ENDPOINT
PGFLEET_S3_BUCKET
PGFLEET_S3_REGION
PGFLEET_S3_ACCESS_KEY
PGFLEET_S3_SECRET_KEY		required Where backups and WAL land. Defaults target the bundled MinIO (localhost:9000, bucket pgbackrest); swap for managed S3 in production.
PGFLEET_INSTANCE_BIND_ADDRESSoptional Host interface that managed instances publish on. Defaults to 127.0.0.1 (private — same-host/proxy access only). Widen to a private IP or 0.0.0.0 only deliberately.
PGFLEET_AUTO_FAILOVERoptional Promote a replica when the primary dies. Defaults to true.

Keep instances private. Leaving PGFLEET_INSTANCE_BIND_ADDRESS at 127.0.0.1 means nothing is reachable from outside the host until you intentionally publish it. Front anything you do expose with a reverse proxy or Cloudflare Tunnel.

4

Log in, create an instance, take a backup

Open the dashboard at http://localhost:3000 and sign in with the bootstrap admin you set above. Then:

  • Create your first instance. Pick a Postgres version (and TimescaleDB if you want hypertables), name it, and provision — PgFleet launches the container and starts archiving WAL to your bucket within seconds.
  • Take a backup. Trigger a full backup from the instance view, or let the schedule (PGFLEET_BACKUP_INTERVAL, default 24h) run it. Backups go to your S3/MinIO bucket via pgBackRest.
  • Prove it with a restore drill. Restore that backup into a fresh instance — or use the PITR wizard to rewind to a chosen second — to confirm it is genuinely restorable. A backup you have never restored is a hope, not a backup.

5

Optional: enable Authelia SSO

By default PgFleet uses its built-in email + password login — no extra setup. Single sign-on is optional and toggled by environment. When you set PGFLEET_SSO_EMAIL_HEADER on the API and NEXT_PUBLIC_OIDC_ENABLED=1 on the dashboard, a “Sign in with SSO” button appears and PgFleet trusts a forward-auth proxy (Authelia or any OIDC proxy) for identity, mapping the user’s groups to PgFleet admin / operator / viewer roles.

.env (API) — enable SSO 
# SSO mounts only when EMAIL_HEADER is set — unset = disabled
PGFLEET_SSO_EMAIL_HEADER=Remote-Email
PGFLEET_SSO_GROUPS_HEADER=Remote-Groups
PGFLEET_SSO_ADMIN_GROUP=pgfleet-admins
PGFLEET_SSO_OPERATOR_GROUP=pgfleet-operators
PGFLEET_SSO_AUTO_PROVISION=true

# dashboard: show the SSO button
NEXT_PUBLIC_OIDC_ENABLED=1

A ready-to-adapt stack — Caddy (TLS + forward_auth) in front of Authelia (login + TOTP MFA + brute-force regulation) in front of PgFleet — lives in deploy/authelia/:

shell 
$ docker compose -f deploy/authelia/docker-compose.yml up -d
# → your domain → Authelia portal → enroll TOTP → dashboard, signed in

Two invariants keep this safe. The proxy must strip client-supplied copies of the trusted headers before forward-auth (the bundled Caddyfile does this), and the API must be reachable only through the proxy — never publish its port directly, or a caller could forge the header. Disabled PgFleet accounts are still rejected at the SSO exchange, so off-boarding revokes access. Full write-up: docs/authelia-sso.md.
